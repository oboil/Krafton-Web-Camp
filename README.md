## 2일차 (2025.08.12)

- jquery를 처음 써 보았다. ajax의 개념을 복습하여 무엇인지 확실히 학습했다.
- jquery 대신 React를 주로 쓰는 추세라고 하여, React만 사용했다. 오늘 작성한 jquery는 매우 간단한 코드였지만, 왜 React나 다른 프레임워크를 사용하게 되었는지 알 것 같았다. jquery만 사용할 때, 경로 정리가 복잡해지고 코드 가독성이 떨어질 것을 예측할 수 있었다. 사실 지금까지 프레임워크의 큰 중요성을 꺠닫지 못했는데, 이렇게 코드를 작성하니 프레임워크가 능률을 높일 수 있다는 것을 깨달았다. 또한 jquery 역시 DOM 전체가 다시 reload되어 일부 state만 업데이트하는 React에 비해 성능이 좋지 않다고 생각한다.

  → 정정: jquery도 일부만 업데이트 되지만, React가 더 최적화되어 있다고 한다. React는 Virtual DOM으로 변경 부분만 업데이트한다.

  ⇒ 이러한 면을 보면, jquery는 간단한 프로젝트에 적용하면 유용할 것 같다.

- ajax로는 주로 API를 호출하여 서버로부터 정보를 가져오는(GET) 것을 했다. 도전 과제로 배틀그라운드의 전적을 가져오는 활동을 했는데, 함수 구현보다는 데이터를 찾는 것이 오래 걸렸다. 알맞는 API를 찾아야 했다. 내가 원하는 정보를 찾고 그것을 가져왔을 때 엄청난 성취감을 느꼈다.
- 백틱(`)의 유용함을 점차 깨닫는 중이다. 변수와 문자열을 혼용하여 쓸 수 있고, 여러 줄을 작성하여 html에 삽입할 때 불편함을 줄일 수 있다. 점점 애용하게 될 것 같다.

## 7일차 (2025.08.17)

- 메인 화면 구현을 담당했다. 메인 화면에서 쓰이는 지도를 구현하는 것이 가장 까다로운 일이었다.

- 첫 번째로, 지역에 온도에 따른 지도 시각화가 필요했다. 우선 지도이므로 네이버 지도나 카카오맵의 API를 사용하려 했는데, 건물 등 상호명까지 나오는 세부적인 지도가 불러졌다. 이는 과도한 정보이며, 우리는 광역시/도와 시/군/구까지의 행정구역만 구분할 것이었기에 적합하지 않다고 생각했다. 무엇보다 행정구역의 경계에 따라 지도에 색을 입히는 것이 어려웠다. 따라서 좌표로 직접 지도를 그리고 시각화를 진행하기로 했다. (좌표 데이터: https://github.com/southkorea/southkorea-maps.git )

- 두 번째로, 행정구역 필터링 문제가 있었다. 광역시/도와 시/군/구 좌표 데이터가 별개의 데이터이므로 확대된 정도에 따라 보이는 지도가 달라야 했다. 하지만 나는 데이터 분석을 잘못하여 시/군/구 데이터로 광역시/도의 지도를 그리려고 하였다. 다시 데이터를 분석한 뒤 각 데이터로 지도를 그리는 것으로 해결했다. 데이터 분석의 중요성을 알 수 있었다.

- 세 번째로, 팀원이 만든 조사 화면을 메인 화면과 연결했다. 이 작업은 붙여넣기만 하고 일부만 수정하면 되었다. 팀원이 HTML을 처음으로 작성하여, 작동에는 큰 문제가 없으나 형식이 맞지 않는다는 점(같은 역할을 하는 텍스트에 h3과 label을 혼용하는 등), 화면별로 클래스명이 통일되지 않았다. 이것은 AI를 이용해 빠르게 통일했다. CSS 파일을 쉽게 수정하기 위해서라도 통일이 필요하다고 생각했다. 프로젝트를 진행할 때 포맷을 정해두는 이유를 깨달았다.

- 네 번째로, 다른 팀원이 만든 기상청 온도, 조사한 온도 데이터를 저장하는 DB와 메인 화면을 연결하는 작업을 했다. 데이터는 문제 없이 주고 받을 수 있었으나 데이터에 따른 시각화에 오류가 있었다. 문제를 분석하니, 좌표 데이터에 있는 지역명과 DB에 있는 지역명이 일치하지 않기 때문이었다. DB에는 작동만 확인하기 위해 모든 지역 데이터가 저장되지 않았기 때문이었다. 따라서 좌표 데이터에 있는 지역과 기상청이 제공하는 실제 행정 구역 데이터를 비교하며 DB에 저장함으로써 해결했다.

- DB의 형태와 불러오는 타이밍을 수정했다. 지금까지는 새로운 사용자가 접속하거나 새로고침할 때마다 기상청에서 온도 데이터를 실시간으로 받아오고 DB에 있는 조사 데이터도 받아오는 구조였다. 접속자가 두 명 이상이 되면 렉이 심했고, 더 많은 사용자가 생기면 DB에 과부하가 올 것으로 예상했다. 따라서 기상청 온도 데이터를 DB에 저장하고, 날씨는 1초 단위로 갱신하는 것이 중요하지 않으니 30분마다 데이터를 갱신하는 것으로 수정했다.
